# Pi Coding Contract

- Evidence first. Use `FACT source`, `HYPOTHESIS confidence + falsifier`, and `UNKNOWN next check`. Never invent missing ticket, code, log, or runtime facts.
- Snapshot repository instructions, branch, `HEAD`, and `git status --short` before non-trivial work. Refresh after repository state changes.
- Route exactly one workflow: ticket implementation, read-only feature investigation, hypothesis-driven bug investigation, or read-only cross-team triage.
- Use Plannotator only for ambiguous, cross-file, or cross-system work. One concise plan and one gate; no automatic pre-plan/post-plan agent passes.
- Delegate only when context isolation helps: zero agents for trivial work, one fresh `scout` normally, at most two for independent evidence streams.
- At code-execution start, resolve the invoking checkout and the stable `PI_SUBAGENT_PARENT_SESSION` key once; require the key to match `[A-Za-z0-9._-]+`. Use branch `pi-session/<key>` and directory `pi-session-<key>` beneath `PI_SUBAGENTS_WORKTREE_DIR`, or beneath the operating-system temporary directory when unset, as the only canonical worktree identity. Create that persistent branch and worktree only when neither exact identity exists; otherwise reuse the registered exact match. Retain and reuse its absolute path for every implementation, review fix, and follow-up enhancement in the session. Never create temporary, child, integration, or replacement worktrees, never delegate with `worktree: true`, and never remove the canonical worktree automatically. If session identity or exact ownership cannot be verified, stop instead of guessing or creating another worktree.
- Use one fresh `worker` as the sole active writer in the canonical session worktree when delegation is useful. Pass bounded briefs and `context: "fresh"`, serialize writers, and use one fresh `reviewer` after meaningful changes. Do not use raw `{previous}` chains or nested agent trees.
- After each reviewed and verified code-changing request, stage only its scoped paths and create exactly one local Conventional Commit in the canonical session worktree. Use `fix:` for patch impact, `feat:` for minor impact, and `<type>!:` plus a `BREAKING CHANGE:` footer for major impact; do not bump versions, tag, or push unless requested. A follow-up enhancement in the same Pi session becomes the next commit in the same worktree.
- Verify subagent claims from actual diff and command output. Completion requires fresh focused checks plus repository-required checks; report anything not run.
- Diagnoses and reviews are read-only by default. Local implementation commits follow the canonical-session rule above. Push, merge request, ticket/document edit, chat reply, production mitigation, version bump, tag, or destructive action requires explicit user request.
- Search files only with `rg` or `rg --files`; never `find`.
- Use Context7 for version-sensitive library/API facts. If unavailable, use current primary docs and state fallback.
- Keep chat terse. Keep evidence, plans, delegation briefs, requirements, and safety steps fully explicit.
