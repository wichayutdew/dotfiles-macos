Workflow: gitlab-mr-comments

Load `caveman` at ultra intensity for chat plus the installed `receiving-code-review` and `coding-standards` skills for triage. Keep plans, evidence, requirements, tests, replies, and safety steps fully explicit.

Stay in Plannotator planning mode until approval. Do not edit code, create a worktree, commit, push, reply, or resolve discussions during planning.

1. Fetch merge request metadata, current head SHA, source/target branches, full diff, and every unresolved discussion. Prefer GitLab MCP; use authenticated `glab` only when MCP is unavailable or incomplete; use direct `curl` last. Never print credentials or secret-bearing commands.
2. Before judging comments, load applicable repository instructions, task-relevant project prompts and skills, architecture/contributor/build docs, branch, HEAD, status, changed code, callers, and tests. Read relevant files fully. Do not switch the planning checkout; if it is not the exact merge-request head, use GitLab content and record the mismatch as evidence. Record unavailable context as `UNKNOWN`.
3. Use one fresh read-only scout for code/history evidence and one fresh reviewer only when independent validation materially helps. Parent verifies each conclusion. No nested delegation.
4. Classify each unresolved comment as valid, partly valid, invalid, or already addressed. Cite causal evidence. For valid comments, propose the smallest code/test change. For pushback, draft a concise evidence-backed reply. Never dismiss feedback from preference alone.
5. Build an annotation-ready plan whose first line is exactly `Workflow: gitlab-mr-comments`, followed by exact headings: `Goal`, `In scope`, `Out of scope`, `Evidence`, `Things to implement`, `Implementation plan`, `Requirement-to-test mapping`, `Skill recommendation`, `Open questions`, and `Risks`. Under `Implementation plan`, use standard `- [ ]` items and include at least one revalidation/report item even when no fix or reply is needed. Include discussion IDs, exact planned fixes, exact draft replies, push target, and checks. Label claims `FACT`, `HYPOTHESIS` with confidence/falsifier, or `UNKNOWN` with next check.
6. In `Skill recommendation`, list relevant installed project/global skills. Recommend a new install only when it materially improves this triage; include source and benefit. Otherwise state `No new skill needed`. Never install during planning.

After approval, re-fetch head SHA and unresolved discussions. Return to planning if evidence changed materially. For approved fixes, reuse or create the session's one canonical worktree only then, basing new work on the exact refreshed merge-request source SHA; delegate code/tests to one fresh worker with the installed Superpowers `test-driven-development` skill plus applicable project skills named in the plan; require red, green, refactor evidence, focused and required wider checks, one fresh reviewer, and one scoped commit. Push without force from canonical HEAD to the verified merge-request source ref, then reply to each approved discussion with what changed and verification. For approved pushback, reply with the approved rationale and no code mutation. Do not resolve discussions, approve, or merge unless the approved plan explicitly includes that separately authorized action.
