Workflow: gitlab-mr-comments

Load `caveman` at ultra intensity for chat plus the installed `receiving-code-review` and `coding-standards` skills for triage. Keep plans, evidence, requirements, tests, replies, and safety steps fully explicit.

Stay in Plannotator planning mode until approval. Do not edit code, create a worktree, commit, push, reply, or resolve discussions during planning.

This workflow remains active until successful `/workflow-done` or explicit `/workflow-abort` without a completion claim. Treat every later user-authored follow-up as a new iteration: re-enter planning, refresh merge-request, discussion, and repository evidence with a new bounded foreground fresh read-only scout, revise the same plan file, and obtain another approval before any new fix, push, or reply. Preserve prior approved work and reuse the canonical worktree. Never act on follow-up scope under an earlier approval. Use `/workflow-status` for a read-only status check and `/workflow-retry` only for a preserved follow-up.

1. Fetch merge request metadata, current head SHA, source/target branches, full diff, and every unresolved discussion. Prefer GitLab MCP; use authenticated `glab` only when MCP is unavailable or incomplete; use direct `curl` last. Never print credentials or secret-bearing commands.
2. Before judging comments, load applicable repository instructions, task-relevant project prompts and skills, architecture/contributor/build docs, branch, HEAD, status, changed code, callers, and tests. Read relevant files fully. Do not switch the planning checkout; if it is not the exact merge-request head, use GitLab content and record the mismatch as evidence. Record unavailable context as `UNKNOWN`.
3. Use one foreground fresh read-only `scout` for code/history evidence. Use one `researcher` only for an independent current external-doc question. Parent verifies each conclusion. No reviewer, planner, delegate, oracle, or nested delegation during planning.
4. Classify each unresolved comment as valid, partly valid, invalid, or already addressed. Cite causal evidence. For valid comments, propose the smallest code/test change. For pushback, draft a concise evidence-backed reply. Never dismiss feedback from preference alone.
5. Build an annotation-ready plan whose first line is exactly `Workflow: gitlab-mr-comments`, followed by exact headings: `Goal`, `In scope`, `Out of scope`, `Evidence`, `Things to implement`, `Implementation plan`, `Requirement-to-test mapping`, `Done when`, `Verification contract`, `Skill recommendation`, `Open questions`, and `Risks`. Under `Implementation plan`, use standard `- [ ]` items and include at least one revalidation/report item even when no fix or reply is needed. On follow-up iterations, add an iteration delta under `Evidence`: corrected requirement, reason for the prior miss, invalidated assumption or evidence, exact regression check, and unfinished work carried forward. Map each delta to an implementation and verification item. Include discussion IDs, exact planned fixes, exact draft replies, push target, and checks. For code fixes, map requirements under `Done when`, then place the exact absolute canonical repository `cwd`, focused commands, complete repository test-suite, check-only format, and lint commands in the machine-readable `Verification contract`: one JSON block with only `cwd`, `worker`, and `reviewer`; every command item has `id`, exact one-line `command`, and positive integer `timeoutMs`; reviewer IDs are exactly `full-tests`, `format`, and `lint`. Never use placeholder success commands; missing commands or `cwd` are an `UNKNOWN` blocker to approval and a passing verdict. For reply-only or no-action plans, write exactly `Not applicable - read-only plan.` under that heading. Label claims `FACT`, `HYPOTHESIS` with confidence/falsifier, or `UNKNOWN` with next check.
6. In `Skill recommendation`, list relevant installed project/global skills. Recommend a new install only when it materially improves this triage; include source and benefit. Otherwise state `No new skill needed`. Never install during planning.

After each approval, re-fetch head SHA and unresolved discussions. Return to planning if evidence changed materially. For approved fixes, reuse or create the session's one canonical worktree only then at the exact contract `cwd`, basing new work on the exact refreshed merge-request source SHA; delegate code/tests to one new fresh `worker` with `context: "fresh"`, that exact `cwd`, the installed Superpowers `test-driven-development` skill plus applicable project skills named in the plan, and runtime verified acceptance with the exact approved worker commands, order, and timeouts and no allowed failures or command overrides. Require red, green, refactor evidence. After a meaningful diff and passed worker ledger, launch one foreground fresh read-only `reviewer` in the same `cwd` with the exact approved runtime commands whose IDs are `full-tests`, `format`, and `lint`, using non-fixing modes with no overrides or allowed failures. Its acceptance `reviewFindings` lists every actionable finding and is empty only when none exist. Any finding returns to one serial worker, then repeats both gates. Do not stage or commit unless the user separately requested it and the approved plan lists it as an exact worker action before the fresh reviewer gate. Push without force from canonical HEAD to the verified merge-request source ref only when the approved plan contains that exact user-authorized push, then post only the approved replies. For approved pushback, reply with the approved rationale and no code mutation. Do not resolve discussions, approve, or merge unless that iteration's approved plan explicitly includes the separately authorized action.
