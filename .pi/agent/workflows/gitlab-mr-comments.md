Workflow: gitlab-mr-comments

Load `caveman` at ultra intensity, `receiving-code-review`, `coding-standards`, and Superpowers `brainstorming`. Keep evidence, plans, tests, proposed replies, and safety steps explicit.

Accept a GitLab, GitHub, GitHub Enterprise, or other HTTPS hosted review URL plus optional user context. Stay in Plannotator planning until approval. Do not edit, commit, push, reply, or resolve threads during planning.

1. Detect host from the URL. An MR/PR URL is the only remote locator; never treat it as a work-item or individual-comment URL. For GitLab, parse the project path and merge-request IID from `.../<project>/-/merge_requests/<iid>`, then read all MR-scoped data with `get_merge_request`, `get_merge_request_commits`, and `get_merge_request_diffs`. Read discussions and notes from that same project and IID with the GitLab MR discussions/notes endpoint or authenticated `glab mr view`/`glab api`; never call `get_workitem_notes`, `create_workitem_note`, or any `workitem` tool for an MR. Read pipeline status with `get_merge_request_pipelines`, then read each pipeline's jobs with `get_pipeline_jobs` using the same project and returned pipeline ID. Fetch description, branches, head SHA, full diff, checks, and every unresolved thread using matching read-only MCP tools first, then authenticated host CLI, then trusted GET/HEAD `curl`. For generic hosts, use available host MCP/CLI or same-host authenticated `curl`. Never cross hosts or print credentials.
2. Explore relevant local or remote code and history with `rg`, `rg --files`, `ast-grep`, read-only Git/CLI, hosted-code MCP, and other read-only tools. Read instructions, changed code, callers, and tests. Use one fresh bounded `scout`.
3. Classify each unresolved comment as valid, partly valid, invalid, or already addressed with causal evidence. Use Superpowers `brainstorming` to create one plan. First line stays `Workflow: gitlab-mr-comments`; use standard workflow headings. Put every acceptance criterion under `Done when`, discussion IDs, exact fixes, exact proposed replies, and exact verification. Add `Remote action contract` containing one JSON object with `actions`; the array may be empty when no remote call is needed. Every proposed action has unique `id`, exact `toolName`, and exact object `input` bound to the review platform, repository, and merge-request or pull-request number.
4. Submit through Plannotator. Revise and resubmit on feedback.
5. A code-fix plan uses exactly one repository entry in the multi-repository `Verification contract` schema from `local-work.md`. Apply the fix in the user's current Git checkout and current branch: both `cwd` and `sourceCwd` are that repository root, `branch` is the branch already checked out, and `baseHead` is its current HEAD. Do not create or switch a worktree or branch. Include the exact Conventional `commitTitle`, acceptance criteria, and verification. Reply-only or pushback plans use `Not applicable - read-only plan.`.

After approval, re-fetch head SHA and unresolved threads. Replan if anchors or scope changed. For a code fix, work only in the user's current checkout and branch, delegate to one fresh `worker`, require Superpowers `test-driven-development`, red-green-refactor evidence, exact verification, all acceptance criteria, and the approved Conventional Commit. Run a fresh reviewer and repeat until clean. Reply-only plans first run the approved read-only scout gate. Then ask the user whether to push and reply by executing every exact unfinished `Remote action contract` entry. Do not push, reply, or resolve before that explicit answer. Each exact call and successful result is ledgered; `/workflow-done` stays blocked until all approved calls finish. A yes never authorizes approval, merge, thread resolution, closure, deletion, or force push.
