Workflow: jira-ticket

Load `caveman` at ultra intensity for chat plus the installed `jira-ticket` and `coding-standards` skills for evidence and scope. Keep plans, evidence, requirements, tests, and safety steps fully explicit.

For code work, require `/ticket <Jira-ticket-ID-or-URL> <rough description>`. Normalize the rough description to lowercase ASCII hyphen-separated text. Use branch `<Jira-ticket-ID>_<rough-description>` and directory `<source-repository-name>-<Jira-ticket-ID>_<rough-description>`.

Stay in Plannotator planning mode until approval. Do not edit code, create a worktree, commit, push, or mutate Jira during planning.

This workflow remains active until successful `/workflow-done` or explicit `/workflow-abort` without a completion claim. Treat every later user-authored follow-up as a new iteration: re-enter planning, refresh the ticket and repository evidence with a new bounded foreground fresh read-only scout, revise the same plan file, and obtain another approval before new implementation. Preserve prior approved work and reuse the canonical worktree. Never execute follow-up scope under an earlier approval. Use `/workflow-status` for a read-only status check and `/workflow-retry` only for a preserved follow-up.

1. You may first conduct bounded read-only context exploration when useful: memory search, repository snapshots, nearest instructions, task-relevant skills and docs, and a bounded scout. This exploration must not make ticket-derived claims or take ticket-driven actions.
2. Fetch the supplied ticket through Atlassian MCP early. Preserve summary, description, acceptance criteria, links, dependencies, status, and relevant comments. Authoritative ticket data is required before making ticket-derived claims, proposing implementation details, or taking ticket-driven actions. If it is unavailable, mark it `UNKNOWN` and stop before those steps.
3. Classify this run as implementation or read-only investigation from explicit ticket/user evidence. Do not blend routes.
4. Before forming hypotheses, inspect each in-scope repository's nearest instructions, task-relevant project prompts and skills, architecture/contributor/build docs, branch, HEAD, status, representative code, callers, and tests. Read relevant files fully. Record sources with `path:line`; never infer coding conventions from language alone.
5. Use one foreground fresh read-only scout for bounded repository evidence. Use one `researcher` only for independent current external facts. Parent verifies every handoff. No nested delegation.
6. Build an annotation-ready plan whose first line is exactly `Workflow: jira-ticket`, followed by exact headings: `Goal`, `In scope`, `Out of scope`, `Evidence`, `Things to implement`, `Implementation plan`, `Requirement-to-test mapping`, `Done when`, `Verification contract`, `Skill recommendation`, `Open questions`, and `Risks`. Under `Implementation plan`, use standard `- [ ]` items and include at least one executable revalidation/report item even for investigation-only outcomes. On follow-up iterations, add an iteration delta under `Evidence`: corrected requirement, reason for the prior miss, invalidated assumption or evidence, exact regression check, and unfinished work carried forward. Map each delta to an implementation and verification item. Label claims `FACT`, `HYPOTHESIS` with confidence/falsifier, or `UNKNOWN` with next check. Include base HEAD for every repository and exact files/symbols when known.
7. In `Skill recommendation`, list relevant installed project/global skills. Recommend a new install only when it materially improves this ticket; include source and benefit. Otherwise state `No new skill needed`. Never install during planning.
8. Map every acceptance criterion to a failing-first check and `Done when` evidence. For implementation, place the exact absolute canonical repository `cwd`, focused commands, complete repository test-suite, check-only format, and lint commands in the machine-readable `Verification contract`: one JSON block with only `cwd`, `worker`, and `reviewer`; every command item has `id`, exact one-line `command`, and positive integer `timeoutMs`; reviewer IDs are exactly `full-tests`, `format`, and `lint`. Never use placeholder success commands. Missing commands or `cwd` are an `UNKNOWN` blocker to approval and a passing verdict. For investigation, write exactly `Not applicable - read-only plan.` under that heading.
9. If work spans multiple Git repositories, present one reviewable ordered repository-session overview. Each session must name its repository root, dependency order, planned branch, base HEAD, acceptance criteria, and verification contract. Support both global defaults and project-local overrides; project-local values take precedence. The overview does not authorize implementation in every repository.

After the overview is approved, require a fresh explicit approval before each repository session. Select or create exactly one isolated worktree only for the approved repository session. Delegate code and tests to one new fresh `worker` with the installed Superpowers `test-driven-development` skill plus applicable project skills named in the plan. The foreground worker launch uses `context: "fresh"`, that exact session `cwd`, and runtime verified acceptance with the exact approved worker commands, order, and timeouts and no allowed failures or command overrides. Require observable red, green, refactor evidence. After a meaningful diff and passed worker ledger, launch one foreground fresh read-only `reviewer` in the same worktree with the exact approved runtime commands whose IDs are `full-tests`, `format`, and `lint`, using non-fixing modes with no overrides or allowed failures. Its acceptance `reviewFindings` lists every actionable finding and is empty only when none exist. Any finding returns to one serial worker, then repeats both gates. A failed or blocked session prevents moving to the next repository. Before moving to the next session, re-enter planning, refresh ticket/repository evidence, revise the same plan, and obtain the next explicit approval. Do not stage or commit unless the user separately requested it and the approved plan lists it as an exact worker action before the fresh reviewer gate. Do not push or create a merge request unless separately authorized. At the end, return the branch name for every repository session.
