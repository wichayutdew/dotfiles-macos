Workflow: gitlab-mr-review

Load `caveman` at ultra intensity, `coding-standards`, `caveman-review`, and Superpowers `brainstorming`. Keep evidence, review findings, plans, proposed comments, and safety steps explicit.

Accept a GitLab, GitHub, GitHub Enterprise, or other HTTPS hosted review URL plus optional user context. This workflow is read-only until the user separately confirms posting comments; it never creates or switches a worktree or branch.

1. Detect host from the URL. An MR/PR URL is the only remote locator; never treat it as a work-item or individual-comment URL. For GitLab, parse the project path and merge-request IID from `.../<project>/-/merge_requests/<iid>`, then read all MR-scoped data with `get_merge_request`, `get_merge_request_commits`, and `get_merge_request_diffs`. Read discussions and notes from that same project and IID with the GitLab MR discussions/notes endpoint or authenticated `glab mr view`/`glab api`; never call `get_workitem_notes`, `create_workitem_note`, or any `workitem` tool for an MR. Read pipeline status with `get_merge_request_pipelines`, then read each pipeline's jobs with `get_pipeline_jobs` using the same project and returned pipeline ID. Fetch description, source/target branches, head SHA, commits, full diff, checks or pipeline, existing discussions, and changed-file context using matching read-only MCP tools first, then authenticated host CLI, then trusted GET/HEAD `curl`. For generic hosts, use available host MCP/CLI or same-host authenticated `curl`. Never cross hosts or expose credentials.
2. Understand proposed changes before judging them. Explore relevant local or remote code, callers, tests, and history with `rg`, `rg --files`, `ast-grep`, read-only Git/CLI, hosted-code MCP, and other read-only exploration tools. Read repository instructions and build docs. Use one fresh bounded `scout`.
3. Use Superpowers `brainstorming` plus evidence to create a review plan. First line stays `Workflow: gitlab-mr-review`; use standard workflow headings. Under `Done when`, list exact review acceptance criteria. Include exact planned comments with current anchors. Use `Not applicable - read-only plan.` under `Verification contract`. Add `Remote action contract` containing one JSON object with `actions`; the array is empty for a clean review. Every proposed comment action has unique `id`, exact `toolName`, and exact object `input` bound to the review platform, repository, and merge-request or pull-request number.
4. Submit through Plannotator. Revise the same plan and resubmit on feedback until approved.

After approval, run one foreground fresh scout with attested acceptance, exact `Done when` criteria, per-criterion evidence, and a structured report. Then ask the user whether to post by executing every exact unfinished `Remote action contract` entry. If yes, re-fetch head SHA and discussions; return to planning if anchors or evidence changed. Execute only exact approved calls and require a successful correlated result for each. Never edit code, create a worktree, approve, merge, resolve threads, close, delete, force-push, or make unlisted remote changes. Stop after every review criterion and posting decision is ledgered.
