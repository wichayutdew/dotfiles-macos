Workflow: gitlab-mr-review

Load `caveman` at ultra intensity for chat plus the installed `coding-standards` skill for review. Keep plans, evidence, requirements, review findings, and safety steps fully explicit.

Stay in Plannotator planning mode until approval. Planning is remote-read-only: do not post comments, approve the merge request, edit code, create a worktree, push, or resolve discussions.

This workflow remains active until successful `/workflow-done` or explicit `/workflow-abort` without a completion claim. Treat every later user-authored follow-up as a new iteration: re-enter planning, refresh merge-request or pull-request and repository evidence with a new bounded foreground fresh read-only scout, revise the same plan file, and obtain another approval before posting any new feedback. Preserve previously approved actions. Never act on follow-up scope under an earlier approval. Use `/workflow-status` for a read-only status check and `/workflow-retry` only for a preserved follow-up.

1. Read `Remote platform` from the workflow kickoff. Fetch merge-request or pull-request metadata, current head SHA, target/source branches, commits, full diff, checks/pipeline state, existing discussions, and changed-file context. For GitLab, prefer configured GitLab MCP, then authenticated `glab`, then direct authenticated `curl`. For GitHub or GitHub Enterprise, use a connected GitHub MCP only when it supports the required operation, then authenticated `gh` (`--hostname <host>` for Enterprise), then direct authenticated `curl` to the trusted API host. Never cross platform, print credentials, or use an untrusted `curl` host.
2. Before reviewing, load applicable repository instructions, task-relevant project prompts and skills, architecture/contributor/build docs, and representative callers/tests from a matching trusted checkout or the selected platform repository. Snapshot branch, HEAD, and status when a local checkout is used. Record unavailable context as `UNKNOWN`.
3. Delegate one foreground fresh read-only `scout` the exact diff plus verified project evidence. Use one `researcher` only for an independent current external-doc question. Parent performs the review judgment and verifies findings against the current diff. No reviewer, planner, delegate, oracle, or nested delegation during planning.
4. Review correctness, edge/error paths, code style against repository conventions, architectural decisions, compatibility, security, concurrency, and measurable performance regressions. Omit praise and speculative nits. Every proposed comment needs current file/line, evidence, impact, and smallest useful fix.
5. Build an annotation-ready plan whose first line is exactly `Workflow: gitlab-mr-review`, followed by exact headings: `Goal`, `In scope`, `Out of scope`, `Evidence`, `Things to implement`, `Implementation plan`, `Requirement-to-test mapping`, `Done when`, `Verification contract`, `Skill recommendation`, `Open questions`, and `Risks`. Under `Implementation plan`, use standard `- [ ]` items and include at least one revalidation/report item even when no comment will be posted. On follow-up iterations, add an iteration delta under `Evidence`: corrected requirement, reason for the prior miss, invalidated assumption or evidence, exact revalidation, and unfinished action carried forward. Map every delta to an action and verification item. Use `Things to implement` for exact draft platform comments. Label claims `FACT`, `HYPOTHESIS` with confidence/falsifier, or `UNKNOWN` with next check. State `No comments` when no actionable findings exist. Under `Done when`, define exact revalidation evidence. Under `Verification contract`, write exactly `Not applicable - read-only plan.` because this route cannot run code workers.
6. In `Skill recommendation`, list relevant installed project/global skills. Recommend a new install only when it materially improves this review; include source and benefit. Otherwise state `No new skill needed`. Never install during planning.

After each approval, re-fetch head SHA and open discussions. If reviewed code or comment anchors changed, return to planning instead of posting stale feedback. Otherwise post only that iteration's approved comments, preserving exact scope and severity. Use only the platform-specific MCP, CLI, or trusted authenticated `curl` route selected above. Do not create a worktree or modify code. Plan approval authorizes only listed comments; it does not authorize approval, merge, resolution, or other remote changes.
